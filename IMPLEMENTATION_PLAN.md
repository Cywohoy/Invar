# Invar 구현 계획

## 목표

Invar는 경쟁 프로그래밍 문제의 입력 Validator를 작성하기 위한 전용
언어다. 입력값의 타입과 제약, 실제 입력 순서, 공백 및 줄바꿈 구조를
Invar 소스로 선언하면 `testlib.h`를 사용하는 C++ Validator 코드를
생성한다.

컴파일러 핵심은 TypeScript로 작성하되 브라우저 API와 Node.js API에
의존하지 않는다. 따라서 같은 핵심 코드를 웹 UI와 향후 Node.js 기반
도구에서 함께 사용할 수 있어야 한다.

확정되지 않은 언어 문법은 구현 과정에서 임의로 추가하지 않는다.

## 전체 처리 구조

```text
Invar 소스
→ Lexer
→ Parser
→ 의미 분석
→ 검증된 중간 표현
→ testlib.h 기반 C++ 코드 생성
```

C++ 컴파일러에 비유하면 각각 토큰화, 구문 분석, 타입 검사, IR 생성,
백엔드에 해당한다.

## 0단계: 최소 언어 사양 확정

구현 전에 현재 예제를 처리하는 최소 언어의 경계를 확정한다.

논의할 항목:

- 식별자 규칙
- 정수 리터럴 규칙
- Invar 소스에서 허용하는 공백과 줄바꿈
- `Int[최솟값..=최댓값]`의 최소 문법
- `String[n]`의 최소 문법
- 선언 순서와 참조 가능 범위
- 중복 선언, 미선언 변수, 사용하지 않은 선언의 처리
- `input` 블록에서 변수의 중복 또는 누락 허용 여부
- `input` 블록 뒤에 공백이나 줄바꿈을 허용할지
- 입력 데이터의 LF와 CRLF 처리
- 생성할 C++ 표준과 사용할 `testlib.h` API
- 초기 오류 보고 범위

배열, 레코드, 여러 테스트 케이스, 일반 refinement, 블록 주석은 당시
최소 단계에서는 설계하지 않고 명시적인 미지원 기능으로 남겼다.

산출물:

- 최소 문법 문서
- 유효한 예제
- 유효하지 않은 예제와 예상 오류
- 미결정 기능 목록

## 1단계: 컴파일러 공통 기반

문법과 무관한 기반을 구현한다.

구현 범위:

- 소스 위치 표현: 오프셋, 줄, 열
- 컴파일 진단 자료구조
- 성공과 실패 결과 표현
- 컴파일러 외부 진입점의 최소 인터페이스
- 브라우저 및 Node.js API 금지 검사 유지

TypeScript의 예외를 내부 제어 흐름에 사용할지, 오류를 결과 값으로
반환할지 결정한다. C++ 관점에서는 `throw`와 `std::expected` 계열의
결과 타입 중 무엇을 사용할지 결정하는 문제와 비슷하다.

테스트:

- 줄과 열 계산
- 빈 소스 처리
- 오류 위치 범위 계산
- 컴파일러 코어의 플랫폼 독립 타입 검사

## 2단계: Lexer

0단계에서 확정한 최소 문법의 토큰만 인식한다.

예상 토큰 범위:

- 식별자와 정수 리터럴
- 현재 예제에 등장하는 키워드
- `:`, `[`, `]`, `..`, `..=`, `<`, `+`, `-`, `;`, `,`, `{`, `}`
- Invar 소스 자체의 공백과 줄바꿈
- 파일 끝
- 잘못된 문자

한 줄 주석은 지원하되 블록 주석이나 문자열 리터럴처럼 아직 결정되지
않은 문법은 추가하지 않는다.

테스트:

- 토큰 종류와 원문 범위
- 여러 자리 정수
- 공백과 줄바꿈
- 부호와 숫자 사이의 소스 공백
- 잘못된 문자
- 파일 끝 위치
- 붙어 있는 기호의 분리

완료 기준은 현재 예제 소스를 정확한 토큰열로 변환하는 것이다.

## 3단계: 재귀 하강 Parser

Lexer의 토큰열을 구문 트리로 변환한다.

구현 범위:

- `val`로 시작하는 단일 및 다중 값 선언
- 현재 확정된 정수 범위 타입
- 현재 확정된 문자열 길이 타입
- 여러 `input` 블록
- `val` 선언과 `input` 블록이 섞인 최상위 항목
- 줄 단위 및 토큰 단위 입력 패턴
- `line(x)` 전체 줄 값 입력 패턴
- trailing comma와 세미콜론이 포함된 토큰 패턴 목록
- 입력 블록 경계를 넘는 미완성 줄 패턴
- 최상위 빈 문
- 예상하지 못한 토큰에 대한 구문 오류

Parser는 문장의 형태만 검사한다. 이름이 실제로 선언되었는지, 입력
순서가 올바른지는 다음 의미 분석 단계에서 검사한다.

테스트:

- 전체 예제 파싱
- 선언별 구문 트리
- 단일 및 여러 `input` 블록 구조
- 빈 `input` 블록
- 닫는 괄호나 세미콜론 누락
- 잘못된 토큰 순서
- `input` 뒤에 다른 선언이 있는 경우
- 빈 줄 입력 블록과 허용되지 않은 빈 입력 위치
- 연속된 세미콜론으로 표현한 여러 빈 줄

## 4단계: 의미 분석

구문상 올바른 프로그램이 실제로 의미 있는지 검사한다.

구현 범위:

- 심볼 테이블 구성
- 중복 선언 검사
- 미선언 이름 참조 검사
- `String[n]`의 길이 의존성 해석
- `input` 블록의 변수 참조 검사
- 토큰 단위 입력 패턴의 단일 대입 검사
- 여러 `input` 블록을 통과하는 값 준비 순서 검사
- 최소 한 개의 `input` 블록 검사
- 의존 대상이 먼저 입력되는지 검사
- 0단계에서 확정한 누락, 중복, 미사용 규칙 검사
- 코드 생성에 적합한 검증된 중간 표현 생성

구문 트리와 검증된 중간 표현을 분리하여 코드 생성기가 잘못된
프로그램을 다시 검사하지 않도록 한다.

테스트:

- 올바른 의존 순서
- `s`가 `n`보다 먼저 입력되는 경우
- 미선언 변수
- 중복 선언
- 입력 블록의 누락과 중복
- 잘못된 타입의 값을 길이로 참조하는 경우
- 문자열 길이의 정적 비음수 증명
- 복구 가능한 여러 진단 수집

## 5단계: C++ 코드 생성

검증된 중간 표현을 `testlib.h` 기반 Validator 코드로 변환한다.

구현 범위:

- 필요한 헤더와 Validator 초기화 코드
- 정수 읽기와 범위 검사
- 대상 testlib 규칙을 따르는 토큰 입력
- 줄바꿈 검사
- 문자열 읽기와 길이 검사
- `String`과 `Int`의 전체 줄 입력
- 입력 파일 끝 검사
- 안정적인 C++ 식별자 생성

구현 전에 논의할 항목:

- 사용할 정확한 testlib API
- 생성 코드의 C++ 표준
- Invar 변수 이름을 그대로 사용할 수 없는 경우의 이름 변환
- Validator 오류 메시지와 변수 설명
- 문자열 길이의 기준
- 마지막 줄바꿈과 파일 끝 처리
- 생성 코드 포맷

테스트:

- 작은 단위별 생성 코드 비교
- 전체 예제의 골든 파일 비교
- 가능한 경우 실제 `testlib.h`와 C++ 컴파일
- 생성한 Validator를 유효하거나 무효인 입력으로 실행하는 통합 테스트

## 6단계: 통합 컴파일 API

각 단계를 하나의 환경 독립적인 API로 연결한다.

개념적인 동작:

```text
compile(source)
→ 성공: 생성된 C++ 소스
→ 실패: 위치 정보가 포함된 진단 목록
```

정확한 TypeScript 인터페이스와 오류 형식은 구현 시점에 확정한다.

테스트:

- 유효한 전체 소스 컴파일
- Lexer 오류 전달
- Parser 오류 전달
- 의미 오류 전달
- 같은 입력에 대해 항상 같은 C++ 출력 생성
- DOM 및 Node.js 없이 실행 가능

여기까지가 Invar 컴파일러 핵심의 첫 번째 완성 지점이다.

### 구현됨: 정수 산술식

최소 컴파일러 다음 확장으로 정수 산술식을 구현했다. 추가된 정수식은
정수 범위 경계와 문자열 길이 등 정수 값을 요구하는 모든 위치에서
동일하게 사용할 수 있다.

지원 범위:

- 단항 `+`, `-`
- 이항 `+`, `-`, `*`, `/`, `%`
- 괄호와 일반적인 산술 우선순위
- Python식 바닥 나눗셈과 나머지
- 상수식의 0 나눗셈 및 64비트 오버플로 컴파일 오류
- 동적 0 나눗셈 및 오버플로를 생성 Validator에서 거부
- 문자열 길이 비음수 증명을 위한 보수적인 구간 분석

### 구현됨: Bool, refinement와 expression 제어 흐름

- `Bool`, `Unit`, `true`, `false`
- 비교, 동등 및 논리 연산자
- `Bool -> Unit`인 `require(...)`
- tail expression으로 값을 결정하는 lexical 블록
- `else`가 필수인 `if` expression
- 같은 기본 타입 계열 안의 refinement 서브타이핑
- 분기 결과의 공통 상위 refinement 계산
- 분기별 definite/possible assignment 흐름 분석
- 블록 지역 선언과 shadowing 금지

### 구현됨: 가변 값, 대입과 반복문

- `var` 선언과 별도 `=` 대입
- 초기화된 `var Int`의 `+=`, `-=`, `*=`, `/=`, `%=`
- 선언 refinement를 보존하는 대입 타입 검사
- refinement expression에서 `var` 현재 값을 선언 시 snapshot
- `for (count) times { ... }` 횟수 반복
- `while (condition) { ... }`
- 반복 횟수의 단일 평가와 음수 검사
- 0회 실행 가능성을 고려한 definite-assignment 분석
- 반복 입력 본문의 줄 경계 보존 검사
- 선택적인 `else`와 Unit 블록 분기로 이루어진 statement 형태 `if`
- statement 형태 `if`, `for`, `while`의 세미콜론 없는 문법

현재 반복문에는 이터레이터, `break`, `continue`가 없다. 조건에 따른
refinement narrowing과 사용자 함수도 후속 설계 대상으로 남긴다.

### 구현됨: 배열과 선언 initializer

- `Array[T, length]`와 중첩 배열
- 선언 시점에 준비된 비음수 길이 검사
- `a[i]` 및 `matrix[i][j]` 원소 접근
- `val` 배열을 포함한 모든 배열의 가변 원소
- 원소 일반 대입과 산술 복합 대입
- `input { a[i]; }` 원소 입력
- 정적 인덱스 오류와 생성 Validator의 동적 경계 검사
- 생성 Validator의 원소 미초기화 읽기 검사
- `Array[String[1], n]`의 압축 토큰 입력
- 단일 `val`/`var` 선언의 `= expression` initializer

배열 전체 줄 입력, 배열 리터럴과 배열용 반복자 문법은 아직 도입하지
않았다. `var`와 배열 원소는 선언 시 snapshot으로 refinement에서
사용할 수 있다. 배열 원소의 동적 경계와 초기화는 생성 Validator가
검사한다.

## 7단계: 최소 웹 UI

컴파일러 핵심이 완성된 뒤 순수 TypeScript로 얇은 웹 계층을 추가한다.

최소 범위:

- Invar 소스 입력 영역
- 컴파일 실행
- C++ 결과 표시
- 오류 위치와 메시지 표시
- 결과 복사
- 전체 줄 번호 표시와 내용에 따라 늘어나는 소스 입력 영역
- 소스 입력 영역과 같은 높이의 C++ 결과 영역
- 생성 C++ 맨 위의 원본 Invar 주석을 켜고 끄는 옵션(기본값 켜짐)

에디터 라이브러리, 문법 강조, 파일 저장 같은 기능은 필요성이 확인되기
전까지 추가하지 않는다.

테스트:

- 컴파일 버튼과 핵심 API 연결
- 성공 결과 표시
- 오류 결과 표시
- 프로덕션 빌드

## 8단계: 배포와 CLI

향후 별도 논의 대상:

- GitHub Pages의 Vite `base` 설정

- 여러 `input` 블록과 단계적 입력 및 검증

## 진행 범위 선택지

진행 범위는 다음 지점 중 하나에서 끊을 수 있다.

1. **설계 단계까지만:** 0단계를 진행하여 최소 언어 사양과 테스트 사례를
   확정한다.
2. **프런트엔드 단계까지:** 0~3단계로 Lexer와 Parser까지 구현한다.
3. **첫 컴파일러 완성까지:** 0~6단계로 현재 예제를 C++ Validator로
   변환한다.

현재 상태에서 가장 안전한 다음 작업은 0단계의 최소 언어 사양
확정이다. 식별자, 소스 공백, 참조 규칙 등을 먼저 결정하지 않으면
Lexer와 Parser가 확정되지 않은 문법을 임의로 선택하게 된다.
